# Steamworks DAST Lab

Steamworks DAST Lab is a controlled vulnerable laboratory designed for the analysis of API security issues in Game-as-a-Service (GaaS) environments inspired by Steamworks-like integrations.

The project provides:

- A vulnerable FastAPI backend.
- A PostgreSQL database.
- A mitmproxy-based DAST observer called Cerberus.
- Automated attack modules.
- Structured JSONL findings.
- A Markdown technical audit report.

The goal is to reproduce, detect and document common API security flaws such as BOLA, BOPLA, weak authentication and business logic abuse in game backend workflows.

## 2. Project Overview

This laboratory simulates a vulnerable game backend with Steamworks-like workflows.

The backend exposes endpoints related to:

- Authentication.
- User profile management.
- Inventory.
- Achievements.
- Cloud saves.
- Microtransactions.

Some flows include both vulnerable and secure implementations. This allows comparing the insecure behavior with the corresponding mitigation.

The project is designed as an academic DAST laboratory. It does not attack real Steam services, real users or production systems.

## 3. Components

### 3.1 Backend

The backend is implemented with FastAPI.

It simulates a Steamworks-like API using synthetic tickets instead of real Steam credentials.

The backend includes:

- Simulated Steam authentication.
- User profile endpoints.
- Inventory endpoints.
- Achievement endpoints.
- Cloud save endpoints.
- Microtransaction endpoints.
- Vulnerable and secure versions of selected flows.

The API documentation is automatically generated by FastAPI and is available through Swagger/OpenAPI.

Default URL:

```text
http://localhost:8000/docs
```

---

### 3.2 Database

PostgreSQL is used to persist laboratory data.

It stores:

- Users.
- Inventory items.
- Achievements.
- Cloud saves.
- Transactions.

The database runs inside Docker Compose.

The database volume can be reset with:

```bash
docker compose -f infra/docker-compose.yml down -v
```

---

### 3.3 Cerberus Proxy

Cerberus is the mitmproxy-based DAST observer of the laboratory.

It is located at:

```text
proxy/addon.py
```

Cerberus observes HTTP traffic flowing through the proxy and stores findings in:

```text
reports/cerberus_findings.jsonl
```

Cerberus currently detects:

- BOLA attempts on inventory endpoints.
- Sensitive client-controlled fields.
- Client-side transaction approval attempts.
- Cross-user resource access by correlating `access_token -> user_id`.

Cerberus does not execute attacks by itself. It passively observes traffic generated by the attack scripts or by any HTTP client configured to use the proxy.

---

### 3.4 Automated Attack Modules

The `attacks/` directory contains Python scripts that reproduce vulnerabilities automatically.

| Script | Purpose |
|---|---|
| `bola_attack.py` | Confirms Broken Object Level Authorization in inventory access. |
| `bopla_attack.py` | Confirms Broken Object Property Level Authorization / Mass Assignment. |
| `weak_token_impersonation.py` | Confirms weak token integrity and user impersonation. |
| `transaction_fraud_attack.py` | Confirms transaction fraud through client-side payment approval. |
| `run_all_attacks.py` | Executes all attack modules. |

The attack scripts generate confirmed findings in:

```text
reports/findings.jsonl
```

The reporting module is located at:

It reads:


The final report combines:

- Confirmed vulnerabilities from active attack modules.
- Observed or confirmed findings from Cerberus.

## 12. Implemented Vulnerabilities

### 12.1 Broken Object Level Authorization

Affected endpoint:

```text
GET /vulnerable/inventory/{user_id}
```

Description:

The vulnerable endpoint allows an authenticated user to request the inventory of another user by modifying the `user_id` path parameter.

Attack flow:

```text
1. Login as attacker.
2. Login as victim.
3. Victim creates an inventory item.
4. Attacker requests /vulnerable/inventory/{victim_user_id}.
5. Backend returns victim inventory.
```

Impact:

An attacker can access inventory resources belonging to another user.

Secure endpoint:

```text
GET /secure/inventory/{user_id}
```

Mitigation:

The secure endpoint checks that the requested `user_id` matches the authenticated `user_id`.

---

### 12.2 Broken Object Property Level Authorization / Mass Assignment

Affected endpoint:

```text
PATCH /vulnerable/users/me
```

Description:

The vulnerable endpoint accepts sensitive fields from the client request body, such as `credits`.

Attack flow:

```text
1. Login as attacker.
2. Send PATCH /vulnerable/users/me with credits = 777777.
3. Backend updates credits.
4. Send the same type of payload to /secure/users/me.
5. Secure endpoint ignores credits and only updates username.
```

Impact:

An attacker can modify internal account properties such as virtual currency.

Secure endpoint:

```text
PATCH /secure/users/me
```

Mitigation:

Use strict input schemas and whitelist only fields that the user is allowed to modify.

---

### 12.3 Weak Token Integrity / User Impersonation

Affected endpoint:

```text
GET /users/me
```

Description:

The demo token is only Base64-encoded JSON. It is not signed, has no expiration and has no integrity protection.

Attack flow:

```text
1. Login as attacker.
2. Login as victim.
3. Forge a token containing the victim user_id.
4. Send GET /users/me with the forged token.
5. Backend returns the victim profile.
```

Impact:

An attacker can impersonate another user by forging a token.

Mitigation:

Use signed JWTs or server-side sessions. In a real Steamworks integration, the server should validate the Steam ticket server-side and then issue a signed session token.

---

### 12.4 Client-Side Trust in Transaction Finalization

Affected endpoint:

```text
POST /vulnerable/transactions/finalize
```

Description:

The vulnerable endpoint trusts the client-controlled field `approved_by_client`.

Attack flow:

```text
1. Login as attacker.
2. Create a transaction.
3. Finalize it with approved_by_client = true.
4. Backend finalizes the transaction.
5. User credits increase without real payment verification.
```

Impact:

An attacker can fraudulently increase virtual currency or complete fake purchases.

Secure endpoint:

```text
POST /secure/transactions/finalize
```

Mitigation:

Never trust client-side payment approval. The backend must verify payment state with Steam Web API or another trusted payment provider.

## 13. Cerberus Detection Logic

Cerberus observes traffic and applies several detection rules.

### 13.1 Session Learning

When Cerberus observes a successful login response from:

```text
POST /auth/login
```

it stores:

```text
access_token -> user_id
```

This enables stateful detection.

---

### 13.2 BOLA Detection

When Cerberus observes:

```text
GET /vulnerable/inventory/{user_id}
```

it compares:

```text
authenticated_user_id
requested_user_id
```

If they differ and the backend responds with HTTP 200, Cerberus registers a confirmed BOLA finding.

---

### 13.3 Sensitive Field Detection

Cerberus flags JSON request bodies containing fields such as:

```text
credits
is_admin
role
approved_by_client
```

These fields are considered sensitive because they should normally be controlled server-side.

This detection is marked as an observation unless an active module confirms real impact.

---

### 13.4 Transaction Approval Detection

Cerberus detects requests to:

```text
POST /vulnerable/transactions/finalize
```

containing:

```json
{
  "approved_by_client": true
}
```

This is marked as a suspicious transaction approval attempt.

## 14. Confirmed vs Observed Findings

The project differentiates between confirmed vulnerabilities and observed suspicious behavior.

### Confirmed Finding

A finding is confirmed when an attack module proves real impact.

Examples:

- Accessing another user's inventory.
- Modifying credits.
- Impersonating another user.
- Increasing credits through fake transaction finalization.

### Observed Finding

A finding is observed when Cerberus detects a suspicious pattern, but the impact is not necessarily confirmed by that proxy rule alone.

Examples:

- A request contains `credits`.
- A request contains `approved_by_client`.
- A request targets a sensitive endpoint.

This distinction helps reduce false positives and makes the audit report more precise.
